# PLAN -- Decentralized Encrypted Chat App (GunDB)

## Vision

Transform this project from a GunDB todo demo into a **decentralized, end-to-end encrypted chat application**. Users self-host a relay server, expose it via a tunnel (ngrok/cloudflare), and share the URL. Anyone with the relay URL and a shared room secret can join and chat securely. The relay server sees only encrypted blobs -- it cannot read any messages.

This is a **BYOS (Bring Your Own Server)** model: no central authority, no accounts, no sign-ups. Just a relay URL, a passphrase, and a display name.

---

## How It Works (User Flow)

```
1. User A runs:  node server/server.js
2. User A exposes the relay via ngrok/cloudflare tunnel
   -> Gets a public URL like https://abc123.ngrok.io/gun
3. User A shares the URL + a room secret with User B
4. Both users open the client app in their browser
5. Both enter:
   - Relay Server URL   (e.g. https://abc123.ngrok.io/gun)
   - Room Secret         (shared passphrase, e.g. "pizza-dragon-42")
   - Display Name        (e.g. "alice")
6. Click "Connect" -> they are in a private, encrypted chat room
7. Messages are encrypted client-side before being sent to Gun
8. The relay forwards encrypted blobs -- it cannot read anything
```

## Encryption Design

### Shared Room Secret Model

All users in a room share the same passphrase. This passphrase is used to derive an encryption key. The model works like a private chatroom -- anyone with the passphrase can read/write messages.

### Cryptographic Flow

```
Input:  roomSecret = "pizza-dragon-42"  (user-entered passphrase)
        relayURL   = "https://abc123.ngrok.io/gun"

Step 1: Derive room salt (deterministic, from relay URL)
        roomSalt = SEA.work(relayURL, "gun-chat-salt")
        -> produces a deterministic hash of the relay URL

Step 2: Derive encryption key (PBKDF2 via SEA.work)
        encryptionKey = SEA.work(roomSecret, roomSalt)
        -> strong key derived from passphrase + salt
        -> all users with the same secret + relay derive the SAME key

Step 3: Derive room ID (so different secrets = different rooms)
        roomId = SEA.work(roomSecret, "room-id-salt")
        -> deterministic hash used as the Gun graph key
        -> relay sees this hash but cannot reverse it to the passphrase

Step 4: Encrypt a message
        encrypted = SEA.encrypt({
          text: "hello!",
          sender: "alice",
          ts: 1710500000000
        }, encryptionKey)
        -> produces an opaque encrypted string

Step 5: Write to Gun
        gun.get('room-' + roomId).set({
          enc: encrypted,
          ts: Date.now()       // unencrypted timestamp for ordering
        })

Step 6: Receiving peer decrypts
        message = SEA.decrypt(encrypted, encryptionKey)
        -> { text: "hello!", sender: "alice", ts: 1710500000000 }
```

### What the relay sees vs. what users see

| Data               | Relay Server        | Users (with secret) |
|--------------------|---------------------|---------------------|
| Message content    | Encrypted blob      | Plaintext           |
| Sender name        | Encrypted (inside)  | Visible             |
| Timestamp          | Visible (metadata)  | Visible             |
| Room ID            | Hashed string       | Knows the secret    |
| Who is connected   | IP/WebSocket info   | See display names   |

## Data Model (Gun Graph Structure)

```
gun
 |
 +-- 'room-<roomIdHash>'           // one node per room
      |
      +-- <auto-generated-soul-1>  // message node
      |    { enc: "SEA{...}", ts: 1710500000000 }
      |
      +-- <auto-generated-soul-2>  // message node
      |    { enc: "SEA{...}", ts: 1710500001000 }
      |
      +-- ...
```

Each message is a child of the room node, created via `.set()` which generates a unique soul (ID). The `enc` field contains the encrypted message payload. The `ts` field is an unencrypted timestamp used for display ordering (this is intentional -- timestamps are needed for sorting and the relay already knows *when* messages flow through it anyway).

   - Each user writes "last read timestamp" to their presence node

    - Client-side enforcement (not cryptographically enforced)

## Security Considerations & Threat Model

### What IS protected
- Message content (encrypted with AES via SEA)
- Sender display names (inside encrypted payload)
- Room secret (never transmitted -- only the derived key is used)

### What is NOT protected
- **Metadata**: The relay knows that connections exist, when messages are sent, and approximate message sizes
- **IP addresses**: The relay sees connecting IPs (mitigated by using Tor/VPN)
- **Room existence**: The relay can see that a room hash exists and has activity
- **Timestamp ordering**: Message timestamps are unencrypted for sorting purposes
- **Brute-force of room secret**: If an attacker captures encrypted messages, they could theoretically try to brute-force the room secret. PBKDF2 (via SEA.work) makes this expensive but not impossible for weak passphrases. Users should choose strong room secrets.

### Trust model
- You trust the relay to **route** messages, not to **read** them
- You trust that your friend doesn't share the room secret with others
- You trust the client code (since it runs in the browser, a compromised client could leak keys)
- The relay operator could theoretically serve a malicious client (mitigated if users build/host the client themselves)

### Recommendations for users
- Use strong room secrets (3+ random words, e.g., "correct-horse-battery")
- Self-host both the relay AND the client for maximum trust
- Use ngrok/cloudflare tunnel with auth if exposing relay to internet
- Don't reuse room secrets across different relay servers

## Quick Reference: GunDB SEA API Used

```js
import Gun from 'gun'
import 'gun/sea'
const SEA = Gun.SEA

// Derive a key from passphrase (PBKDF2)
const key = await SEA.work(data, salt)

// Encrypt data with a key
const encrypted = await SEA.encrypt(data, key)

// Decrypt data with a key
const decrypted = await SEA.decrypt(encrypted, key)

// Sign data (not needed for shared-secret model, but available)
const signed = await SEA.sign(data, pair)

// Verify signature
const verified = await SEA.verify(signed, pair.pub)

// Generate keypair (for future per-user auth)
const pair = await SEA.pair()

// Derive shared secret between two users (for future ECDH)
const secret = await SEA.secret(otherUser.epub, myPair)
```
